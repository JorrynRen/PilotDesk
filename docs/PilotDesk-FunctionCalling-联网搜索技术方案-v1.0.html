PilotDesk Function Calling 与联网搜索
技术实现方案

版本 v1.0 | 2026-06-01 | 方案评审

PilotDesk Tauri 2.0 中高难度 预估 18-23 工时

1. 背景分析

1.1 现状

PilotDesk 当前通过 apiClient.ts 与大模型 API 直连，支持 OpenAI 兼容格式的对话接口。消息流经 useWebSocket.ts 中的 sendApiChat 函数，构建 HTTP 请求并通过 ReadableStream 实时接收 SSE 响应。

该架构实现了基本的对话能力，但存在两个核心功能缺失：

  • 深度推理：无法展示模型的思考过程（如 DeepSeek 的 reasoning_content 字段），用户无法了解模型推理链
  • 联网搜索：模型无法获取实时信息，无法回答时效性问题，无法检索互联网数据

1.2 问题根源

当前 sendApiChat 仅处理 choices[0].delta.content 文本流，未解析 reasoning_content 推理内容字段；同时请求体中未携带 tools 参数，模型无法触发工具调用流程。

1.3 目标

功能描述优先级
推理可视化解析并渲染 reasoning_content 字段，以可折叠面板展示思考过程P0
联网搜索通过 Function Calling 协议让模型自主调用搜索工具获取实时信息P0
通用工具系统可扩展的工具注册机制，为未来添加更多工具（文件操作、代码执行等）预留接口P1

2. 方案选型

2.1 联网搜索实现方案对比

维度 方案 A：标准 Function Calling 方案 B：预检索注入 方案 C：Sidecar 代理
原理 在 API 请求中声明 tools，模型自主判断何时调用搜索 在发送前先执行搜索，将结果注入 system prompt 所有请求经 Sidecar 中转，由 Sidecar 处理搜索
搜索时机 模型自主判断（准确率高） 每次对话都搜索（大量冗余） Sidecar 代理判断（增加延迟）
架构侵入 低（改 sendApiChat + 新增工具模块） 低（改 handleSend 前置搜索） 高（需重构通信架构）
可扩展性 高（通用 tools 协议） 低（硬编码搜索逻辑） 中（Sidecar 统一管理）
API 兼容性 OpenAI 兼容协议，主流模型支持 无特殊要求 无特殊要求
对现有 API 直连的影响 无影响（tools 可选参数） 增加延迟（前置搜索） 破坏直连优势

决策 1：推荐方案 A（标准 Function Calling）

理由：OpenAI 兼容的 tools 协议已成为事实标准，模型自主判断搜索时机的准确率高；改动集中在 sendApiChat 不引入新进程依赖；保持 API 直连模式的核心优势；可扩展为通用工具系统。

2.2 搜索引擎选型

引擎类型费用优势劣势
SearXNG自部署元搜索免费聚合多引擎、JSON API 简洁、可私有部署需自行部署维护
TavilySaaS 搜索 API免费额度 / 付费专为 AI Agent 设计、返回结构化结果依赖第三方、有用量限制
DuckDuckGo公开搜索免费无需注册、隐私友好无官方 API、接口不稳定

决策 2：搜索引擎首选 SearXNG，备选 Tavily

理由：SearXNG 免费自部署、聚合多搜索引擎保障结果质量、JSON API 简洁易集成。Tavily 作为备选，适合不想自部署的场景。

2.3 工具循环执行位置

决策 3：工具循环在前端（useWebSocket）而非 Sidecar

理由：保持 API 直连模式不依赖 Sidecar 的核心优势；工具调用仅需 HTTP 请求搜索引擎，无需 Sidecar 中转；前端可直接控制循环次数和 UI 反馈。

4.3 搜索引擎模块（searchProvider.ts）

// 搜索引擎抽象接口
interface SearchProvider {
  name: string;
  search(query: string, maxResults?: number): Promise<SearchResult[]>;
}

interface SearchResult {
  title: string;
  url: string;
  snippet: string;
}

// SearXNG 实现
class SearXNGProvider implements SearchProvider {
  constructor(private baseUrl: string, private apiKey?: string) {}

  async search(query: string, maxResults = 5): Promise<SearchResult[]> {
    const url = `${this.baseUrl}/search?q=${encodeURIComponent(query)}
                 &format=json&categories=general`;
    const resp = await fetch(url, {
      headers: this.apiKey ? { 'Authorization': `Bearer ${this.apiKey}` } : {}
    });
    const data = await resp.json();
    return data.results?.slice(0, maxResults).map((r: any) => ({
      title: r.title,
      url: r.url,
      snippet: r.content
    })) ?? [];
  }
}

4.4 工具执行器（toolExecutor.ts）

// 内置 web_search 工具定义
const WEB_SEARCH_TOOL: ToolDefinition = {
  type: 'function',
  function: {
    name: 'web_search',
    description: '搜索互联网获取最新信息。当用户询问时效性内容、'
                + '最新新闻、实时数据或你不确定的事实时调用此工具。',
    parameters: {
      type: 'object',
      properties: {
        query: {
          type: 'string',
          description: '搜索关键词'
        }
      },
      required: ['query']
    }
  }
};

// 工具执行器
class ToolExecutor {
  private providers = new Map<string, SearchProvider>();
  private tools = new Map<string, ToolDefinition>();

  constructor() {
    // 注册内置工具
    this.registerTool(WEB_SEARCH_TOOL);
  }

  registerTool(tool: ToolDefinition) {
    this.tools.set(tool.function.name, tool);
  }

  setSearchProvider(provider: SearchProvider) {
    this.providers.set(provider.name, provider);
  }

  async execute(toolCall: ToolCall): Promise<ToolResult> {
    const { name, arguments: argsStr } = toolCall.function;
    const args = JSON.parse(argsStr);

    if (name === 'web_search') {
      const provider = this.getPrimaryProvider();
      const results = await provider.search(args.query, 5);
      const content = results.map(r =>
        `【${r.title}】(${r.url})\n${r.snippet}`
      ).join('\n\n');
      return { toolCallId: toolCall.id, toolName: name, content };
    }

    return { toolCallId: toolCall.id, toolName: name,
             content: `未知工具: ${name}`, isError: true };
  }
}

4.5 sendApiChat 工具循环改造

// useWebSocket.ts 中 sendApiChat 核心改造
async function sendApiChat(params: {
  messages: ChatMessage[];
  apiProviderId: number;
  apiModel: string;
  onChunk: (content: string) => void;
  onReasoning: (content: string) => void;  // 新增
  onToolCalls: (toolCalls: ToolCall[]) => void; // 新增
  onToolResult: (result: ToolResult) => void;  // 新增
  onDone: (fullContent: string) => void;
  onError: (err: string) => void;
}) {
  const MAX_TOOL_ROUNDS = 5;
  let round = 0;
  let messages = [...params.messages];
  let finalContent = '';

  // 获取工具定义（从设置中判断是否启用搜索）
  const tools = getEnabledTools();
  const toolChoice = tools.length ? 'auto' : undefined;

  while (round < MAX_TOOL_ROUNDS) {
    round++;
    let assistantContent = '';
    let reasoningContent = '';
    let toolCalls: ToolCall[] = [];

    const body = buildBody({
      messages, model: params.apiModel,
      stream: true, tools, toolChoice,
      // ... 其他参数
    });

    // SSE 流式解析
    await sendApiRequest(url, headers, body, {
      onChunk: (s) => { assistantContent += s; params.onChunk(s); },
      onReasoning: (s) => { reasoningContent += s; params.onReasoning(s); },
      onToolCallChunk: (tc) => { /* 累积 tool_calls */ },
    });

    // 判断是否有工具调用
    if (toolCalls.length > 0) {
      // 添加 assistant 消息（含 tool_calls）到消息数组
      messages.push({
        role: 'assistant', content: assistantContent,
        reasoningContent, toolCalls
      });

      // 执行每个工具调用
      for (const tc of toolCalls) {
        const result = await toolExecutor.execute(tc);
        params.onToolResult(result);
        // 添加 tool 结果消息
        messages.push({
          role: 'tool', content: result.content,
          toolCallId: result.toolCallId
        });
      }
      // 继续循环，让模型处理工具结果
      continue;
    }

    // 无工具调用，最终回复
    finalContent = assistantContent;
    break;
  }

  params.onDone(finalContent);
}

9. 风险与应对

风险影响概率应对措施
部分模型不支持 tools 参数 工具调用失败或被忽略 中 tools 为可选参数，不传时退化为普通对话；在 UI 中标注模型兼容性
搜索引擎服务不可用 搜索工具执行失败 中 try-catch 包装 + 超时机制 + 错误信息回传模型继续对话
工具循环无限死循环 API 调用次数失控 低 硬限制最大轮数（默认 5）+ 单次会话总调用计数限制
reasoning_content 格式不统一 解析失败或显示异常 中 JSON 和文本两种格式兼容解析；缺失时静默不显示
tool_calls 在流式传输中分片 工具调用参数不完整 中 累积式解析：维护 toolCallBuffer 按 index 累积每个工具调用的参数片段

10. 设计决策记录

决策 1：采用标准 Function Calling 协议

OpenAI 兼容的 tools 协议已成为事实标准，DeepSeek、Qwen、Claude 等主流模型均已支持。模型自主判断搜索时机准确率高，避免了方案 B 的冗余搜索和方案 C 的架构侵入。

决策 2：搜索引擎首选 SearXNG

免费自部署不依赖第三方、聚合 Google/Bing/DuckDuckGo 等多引擎保障结果质量、JSON API 简洁。Tavily 作为备选适合不想自部署的用户。

决策 3：工具循环在前端（useWebSocket）

保持 API 直连模式是 PilotDesk 的核心架构优势，将工具循环放在前端避免了对 Sidecar 的依赖，同时前端可以直接控制 UI 反馈（搜索中/分析中状态）。

决策 4：工具系统可扩展设计

toolExecutor 采用注册机制，内置 web_search 工具，同时预留接口用于未来扩展文件操作、代码执行等工具，为 Agent 能力演进奠定基础。

决策 5：推理内容存储到数据库

reasoning_content 存入 messages 表而非仅内存中，确保刷新页面后仍可查看历史推理过程，同时为后续分析（推理质量评估）提供数据基础。
